Problem Definition

플랫폼 API 한계와
비즈니스 리스크

마스터 API 키가 조회할 수 있는 데이터셋이 100개로 수치 제약됨에 따라, 과거 데이터가 API 리포트에서 소실되는 심각한 인프라 관리 장애가 존재했습니다.

API Visibility Limit 100
수작업 권한 부여 공수 5m / 건

조회 실종(Invisible) 장애 현상

새로운 3D 데이터가 적재될 때마다 마스터 인덱스 100 바깥으로 밀려난 과거 라벨 데이터에 대한 제어권이 유실되어, 학습 데이터셋 전처리 자동화 파이프라인에서 제외되는 품질 장애 리스크가 발생합니다.

해결 방향: 동적 다중 계정 배분

마스터 슬롯 임계치를 동적으로 예측하고, 소실 임박한 꼬리(Tail) 데이터셋에 협업 서브 계정을 매핑시켜 조회가 차단되는 현상을 인프라 비용 없이 완벽히 방어합니다.

Architecture

위임 및 업로드
파이프라인 아키텍처

로컬에서 중복 검사를 끝낸 뒤, 마스터 용량 임계치(100개)가 초과될 때 꼬리 데이터셋을 분기하여 서브 계정(Worker) 관리자로 이관하는 지능형 스케줄링 흐름입니다.

  • 1. Slot 계산: 업로드 예정 수량과 잔여 조회 가능량 분석
  • 2. 꼬리 판별: 생성일자 기준 실종 우선순위 N개 추출
  • 3. 협업 위임: 서브 계정에 Collaborator 권한 주입
  • 4. 보정 전송: ego_pose 축 설정 및 default_z 가공 업로드
delegator.py 
def secure_tail_accessibility(upload_count=1):
    expected_total = current_count + upload_count
    if expected_total <= VISIBILITY_LIMIT:
        return
    
    # Identify datasets about to fall off the Master's list
    overlap_n = expected_total - VISIBILITY_LIMIT
    tail_candidates = master_datasets[-min(overlap_n, current_count):]
    
    for ds in tail_candidates:
        # Assign worker with remaining visibility slots
        master_client.add_dataset_collaborator(
            ds.full_name, worker_username, role="manager"
        )
